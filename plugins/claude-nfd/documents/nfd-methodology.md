## 概要

NFDは、AIエージェントを「書く（Write）」のではなく「育てる（Nurture）」開発手法。従来の Code-First（決定論的パイプライン）や Prompt-First（静的なシステムプロンプト）では捉えきれない、専門知識の3つの特性に対応する:

1. **暗黙性（Tacit）** — 状況に直面して初めて発現する判断基準
2. **個人性（Personal）** — 専門家固有のアプローチ
3. **進化性（Evolving）** — 市場や技術に連動して常に変化する動的な性質

## 3層の認知アーキテクチャ

知識を揮発性と個人化の度合いで階層化する:

| 層 | 役割 | 更新頻度 | 管理形態 |
|----|------|----------|----------|
| 憲法層 (Constitutional) | アイデンティティと根本原則 | 低（恒久的） | 役割定義、倫理、品質基準 |
| スキル層 (Skill) | タスク特化型の具体的機能 | 中（月次〜） | モジュール式管理（.md単位） |
| 経験層 (Experiential) | 運用記録とケースメモリ | 高（日次） | セマンティック検索対象 |

### 情報の流動性メカニズム

```
    ┌─────────────┐
    │  憲法層      │  ← 恒久的な原則
    └──────┬──────┘
       ↑結晶化  ↓グラウンディング
    ┌──────┴──────┐
    │  スキル層    │  ← モジュール式
    └──────┬──────┘
       ↑結晶化  ↓グラウンディング
    ┌──────┴──────┐
    │  経験層      │  ← 日々の蓄積
    │  ┄┄共有メモリ┄┄│
    └─────────────┘
```

- **グラウンディング（上→下）**: 憲法層・スキル層を土台として業務入力を解釈する
- **結晶化（下→上）**: 経験層の断片からパターンを抽出し、スキル層・憲法層へ昇華する
- **共有メモリによる協調**: 各スキルモジュールを共有された経験メモリを介して連携させる

## ナレッジ・クリスタリゼーション・サイクル（KCC）

NFDの核心プロセス。日々の断片的な対話ログを再利用可能な知識資産へ変換する。

### 経験的蓄積の6カテゴリ

| # | カテゴリ | タグ | 記録する内容 |
|---|----------|------|-------------|
| 1 | 操作記録 | `[operational]` | 決定とアクションの事実 |
| 2 | 推論追跡 | `[reasoning]` | 判断の背後にあるロジックと前提条件 |
| 3 | パターン観察 | `[pattern]` | 繰り返し発生する事象の共通点 |
| 4 | 失敗原因 | `[error]` | 間違いの本質的理由と修正手順 |
| 5 | 文脈注釈 | `[context]` | プロジェクト背景やチーム固有の方針 |
| 6 | 洞察断片 | `[insight]` | 未整理だが将来重要になり得る気づき |

### 結晶化の5ステップ

1. **[EXTRACT]** — 蓄積されたログから再利用可能なテーマをマイニングし、共通パターンを特定する
2. **[STRUCTURE]** — 抽出した情報をモジュール式のスキル形式に整理する
3. **[DE-CONTEXTUALIZE]** — 特定の事例から固有名詞や状況を除去し、汎用的な原則に抽象化する
4. **[VALIDATE]** — 専門家による検証。過学習や論理的矛盾がないか確認する
5. **[INTEGRATE]** — 検証済みスキルをバージョン管理下に置き、スキル層へマージする

### プラグインの8ステップとの対応

プラグインの8ステップは、論文の5ステップを Claude Code の実装に適用する際に、前処理（Step 1-2）と成果物出力（Step 6）を明示的なステップとして分離したもの:

| 論文5ステップ | プラグイン8ステップ |
|-------------|------------------|
| （前処理） | Step 1: 設定読み込みと位置確認 |
| （前処理） | Step 2: データソースの収集 |
| EXTRACT | Step 3: 並列抽出（サブエージェント） |
| STRUCTURE | Step 4: 統合と重複排除 |
| DE-CONTEXTUALIZE | Step 5: 過学習防止ガード + 脱文脈化 |
| VALIDATE | Step 5（AI検証） + Step 8（人間検証） |
| （成果物出力） | Step 6: 結晶化レポート出力 |
| INTEGRATE | Step 7: ログ更新とGit commit |
| （報告） | Step 8: 結果報告と人間検証 |

## デュアル・ワークスペース・パターン

運用環境と分析環境を論理的に分離し、認知干渉を防止する:

- **ナーチャリング・ワークスペース（日常運用）** — 業務を遂行し経験を蓄積する場
- **サージカル・ワークスペース（外科的修正）** — データ分析、リファクタリング、結晶化を行う場

### 分離の本質: 認知コンテキストの分離

**デュアルワークスペースの「分離」とは認知コンテキストの分離であり、ファイルシステムの分離ではない。**

日常の開発セッションでは、開発者とAIエージェントは「今取り組んでいるタスク」に最適化された認知状態にある。結晶化プロセスは大量のログデータを読み込み、横断的にパターンを抽出する全く異なる認知モードを要求する。この2つを同一セッションで混在させると:

- コンテキストウィンドウがデータ分析に消費され、本来の作業コンテキストが圧迫される
- 日常作業で蓄積した暗黙の前提がパターン抽出にバイアスをかける
- 結晶化の途中結果が日常の判断に干渉する

したがって分離すべきは **セッション（認知スコープ）** であり、ストレージ（ファイルシステム）ではない。auto memory やNFDトレースといった共有リソースは、両ワークスペースからアクセスされることが設計上想定されている。並行書き込みの競合にのみ注意すればよい。

> **よくある誤解**: 「別ディレクトリで作業する」「git worktree でファイルを分離する」ことがデュアルワークスペースの本質だと思われがちだが、それは手段の一つに過ぎない。最もシンプルな実現方法は「別のターミナルで新しいセッションを開く」ことであり、同じディレクトリで作業しても認知コンテキストは分離される。

### Claude Codeでの実現方法

1. **新規セッション**（推奨・最もシンプル）: 別ターミナルで `claude` を起動し、結晶化を実行する。同じディレクトリで構わない — 分離されるのはセッション（認知コンテキスト）であり、ファイルシステムではない
2. **git worktree**（より厳密な分離）: `git worktree add` で親プロジェクトの作業コピーを分離し、そこで結晶化を実行する。ファイルの編集競合も防ぎたい場合に有効

### 関連する分離: NFDリポジトリの独立性

NFDディレクトリ（デフォルト: `.nfd/`、`/nfd-init` で変更可能）が独立Gitリポジトリであることは **データの分離**（NFDの経験データを親リポジトリに混ぜない）が目的であり、上記のワークスペース分離（認知干渉の防止）とは異なる関心事である。両者を混同しないこと。

### マージの流れ

週次〜月次を目安に、サージカルワークスペースの成果（結晶化レポート）を確認し、有用なパターンをナーチャリングワークスペース側のルール・スキルに手動統合する。NFDリポジトリは独立Gitリポジトリのため、`git -C <nfd-dir> log` で履歴確認、`git -C <nfd-dir> revert` でロールバック可能。

## Claude Codeへのマッピング

| NFDの概念 | Claude Codeでの実装 |
|-----------|-------------------|
| 憲法層 | `CLAUDE.md` + `.claude/rules/` |
| スキル層 | `.claude/skills/`（モジュール式） |
| 経験層 | セッションログ（JSONL） + auto memory + NFDトレース（`<nfd-dir>/traces/`） |
| KCC | `crystallize` スキル（本プラグイン） |
| デュアルワークスペース | 新規セッション / git worktree での結晶化実行 |
| 経験の蓄積 | セッションログ（JSONL） + auto memory + NFDトレース + オプショナルMCPツール |
