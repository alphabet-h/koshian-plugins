# Knowledge Crystallization（知識の結晶化）

NFD（Nurture-First Development）の核心プロセス。日常の開発経験から蓄積された断片的なデータを分析し、再利用可能な構造化知識（パターン・ルール・スキル）に変換する。

## 前提条件

- プロジェクトにNFD基盤が構築済みであること（`/nfd-init` で構築）
- NFDの設定ファイル（`config.md`）でデータソースが設定済みであること

## サージカルワークスペースでの実行（推奨）

NFDのデュアルワークスペース・パターンに従い、結晶化は **日常の作業セッションとは別のセッション** で実行することを推奨する。日常の作業中に結晶化を実行すると、大量のデータがコンテキストに流入し、AIの推論に認知干渉を引き起こす可能性がある。

**推奨される実行方法:**

1. **新規セッションで実行**（最も簡単）: 新しいターミナルで `claude` を起動し、`/crystallize` を実行する
2. **git worktree で実行**（より厳密な分離）: `git worktree add ../<project>-nfd nfd-work` で親プロジェクトの作業コピーを分離し、そこで結晶化を実行する。日常作業の作業ツリーに一切影響を与えない

**結晶化後のマージ手順（週次〜月次が目安）:**

1. 結晶化レポート（`crystals/YYYY-MM-DD-crystallization.md`）を確認する
2. 統合したいパターンを選定する
3. 日常の作業セッションに戻り、選定したパターンを統合する:
   - ルール化: `.claude/rules/<rule-name>.md` にパターンの原則と適用条件を記述
   - スキル化: `.claude/skills/<skill-name>/SKILL.md` に手順とガイダンスを記述
4. 統合後、動作を確認してプロジェクトリポジトリにコミットする

結晶化セッションの成果はNFDリポジトリに自動コミットされる。ロールバックが必要な場合は `git -C <nfd-dir> log` で履歴確認、`git -C <nfd-dir> revert <commit>` で戻せる。

## 結晶化の8ステップ

### Step 1: 設定の読み込みと位置確認

1. NFD設定ファイル（`config.md`）を読み込み、YAMLフロントマターからデータソースのパスを取得する
2. `crystallization-log.md` から前回の処理位置を確認する
3. インクリメンタル処理のため、前回の処理位置以降のデータのみを対象とする

設定ファイルが見つからない場合は、プロジェクトルートから以下の順で探索する:
1. `.nfd/config.md`（デフォルトパス）
2. Glob `**/config.md` で `nfd_dir` フロントマターを持つファイルを探索

見つかった `config.md` の `nfd_dir` の値を `<nfd-dir>` として、以降のすべてのパスに使用する。

### Step 2: データソースの判定と収集

設定されたデータソースから入力データを収集する:

| ソース | 収集方法 | 未処理の判定基準 |
|--------|----------|-----------------|
| **NFDトレース** (`traces`) | `traces/` 内のファイルを Read | ログに記録された最終処理ファイル名より新しいファイル |
| **auto memory** (`auto_memory`) | `MEMORY.md` と `~/.claude/projects/<project-hash>/memory/` を Read | MEMORY.md の最終更新日時がログの記録より新しい場合 |
| **セッションログ** (`session_logs`) | `~/.claude/projects/<project-hash>/*.jsonl` を Glob し Read | ログに記録された最終処理JSONLファイル名より新しいファイル |
| **セッションデータ** (`session_data`) | claude-mem MCP の `smart_search` で検索 | ログの最終処理日以降のデータ |
| **カスタムソース** (`custom`) | パスに応じて Read / Glob | ログの最終処理日以降に更新されたファイル |

**セッションログの読み込み制限:** JSONLファイルは大量のデータを含むため、以下の制限を適用する:
- 未処理のJSONLファイルのみを対象とする（ファイル更新日時で判定）
- 各ファイルは末尾2000行を上限として Read（offset/limit パラメータを使用）
- 全データソースの合計読み込み量がコンテキストを圧迫する場合は、直近のファイルを優先し古いものを省略する

**重要**:
- auto memory とセッションログは Claude Code 標準機能のため、ほとんどのプロジェクトで利用可能。初回の結晶化で最も有用なデータソースとなる
- claude-mem MCP が利用できない場合は、セッションデータの分析をスキップし、他のデータソースのみで処理を続行する。エラーにはしない

データが一切見つからない場合は、その旨を報告して終了する。

### Step 3: 並列抽出（サブエージェント）

収集したデータが十分にある場合（複数ソースがある場合）、Agent ツールで最大3つのサブエージェントを並列起動し、各データソースを分析する:

**各サブエージェントの指示テンプレート:**

```
以下のデータを分析し、NFDの6カテゴリで分類してパターンを抽出せよ。

## 6カテゴリ
- [operational] 操作記録: 決定とアクションの事実
- [reasoning] 推論追跡: 判断の背後にあるロジックと前提条件
- [pattern] パターン観察: 繰り返し発生する事象の共通点
- [error] 失敗原因: 間違いの本質的理由と修正手順
- [context] 文脈注釈: プロジェクト背景やチーム固有の方針
- [insight] 洞察断片: 未整理だが将来重要になり得る気づき

## 出力形式
各パターンについて以下を記述:
1. カテゴリタグ
2. パターン名（簡潔に）
3. 観察内容（具体的な事例を引用）
4. 抽象化した原則（プロジェクト固有の情報を除去。社名・プロダクト名・人名・特定URL等の固有名詞は汎用的な表現に置換すること）
5. 証拠の強さ（単一事例/複数証拠/確立済み）
6. 推奨アクション（ルール化/スキル化/経過観察）

## 分析対象データ
<ここにデータソースの内容を挿入>
```

データソースが1つしかない場合は、サブエージェントを使わず直接分析する。

### Step 4: 統合と重複排除

各サブエージェントの結果（または単一分析の結果）を統合する:

1. 同一または類似のパターンをマージする
2. 複数ソースで確認されたパターンの証拠強度を引き上げる
3. 証拠の強さでソートする（確立済み > 複数証拠 > 単一事例）

### Step 5: 過学習防止ガードと脱文脈化

以下のチェックを適用し、結果をフィルタリングする:

- **単一事例ルール**: 1件の事例のみから導出されたパターンには `[要経過観察]` タグを付与し、即座にルール/スキル化しない
- **文脈依存チェック**: 特定のライブラリ/バージョン/環境に強く依存するパターンは、その制約を明記する
- **矛盾チェック**: 既存のルール/スキルと矛盾するパターンがあればフラグを立てる
- **脱文脈化チェック**: パターンの記述にプロジェクト固有の名称（社名、プロダクト名、人名、特定URL、内部システム名等）が含まれていないか確認し、含まれている場合は汎用的な表現に置換する。ただし、脱文脈化により元の知見の意味が損なわれないよう注意する

### Step 6: 結晶化レポートの出力

`crystals/` ディレクトリに結果ファイルを出力する:

**ファイル名**: `YYYY-MM-DD-crystallization.md`

**フォーマット**:

```markdown
# 結晶化レポート — YYYY-MM-DD

## 概要
- 分析データソース: （使用したソースを列挙）
- 抽出パターン数: N
- うちルール/スキル化推奨: M
- うち要経過観察: K

## パターン一覧

### 1. [カテゴリタグ] パターン名

- **観察**: 具体的な事例
- **原則**: 抽象化した知見
- **証拠強度**: 単一事例 / 複数証拠 / 確立済み
- **推奨アクション**: ルール化 / スキル化 / 経過観察
- **統合先候補**: （該当する場合、既存のルール/スキルのパス）

## 推奨アクション一覧

### ルール化候補
（ここに `.claude/rules/` への追加を推奨するパターンを列挙）

### スキル化候補
（ここに `.claude/skills/` への追加を推奨するパターンを列挙）

### 経過観察
（ここに次回の結晶化まで観察を続けるパターンを列挙）
```

### Step 7: ログ更新とGit commit

1. `crystallization-log.md` の処理位置テーブルを更新する:

```markdown
| データソース | 最終処理位置 | 最終処理日 |
|-------------|-------------|-----------|
| traces | <最終処理ファイル名> | YYYY-MM-DD |
| auto_memory | <MEMORY.md最終更新日時> | YYYY-MM-DD |
| session_logs | <最終処理JSONLファイル名> | YYYY-MM-DD |
| custom:<名前> | <最終処理ファイル名/日時> | YYYY-MM-DD |
```

2. 実行履歴を追記する
3. NFDリポジトリにコミットする:

```bash
git -C "<nfd-dir>" add -A
git -C "<nfd-dir>" commit -m "Crystallization: YYYY-MM-DD — N patterns extracted"
```

**重要**: `git -C` でNFDディレクトリを明示的に指定し、親リポジトリにコミットしないこと。`cd` は使用しない。

### Step 8: 結果報告と人間検証

ユーザーに以下を報告する:

1. **抽出されたパターンの概要**（数と主要なもの3〜5件のサマリー）
2. **推奨アクション**（ルール化/スキル化すべきものをハイライト）
3. **次のステップ**: 「結晶化レポート（`crystals/YYYY-MM-DD-crystallization.md`）を確認し、統合したいパターンがあれば指示してください」

**人間検証の必須化**: 結晶化の結果を自動でルール/スキルに統合してはならない。必ずユーザーの確認と明示的な指示を待つこと。

**人間検証のチェックポイント（ユーザーに提示する）:**
- このパターンは自分の経験と合致するか？
- 過度に一般化されていないか？逆に特殊すぎないか？
- 既存のルール/スキルと矛盾しないか？
- このパターンをルール/スキル化した場合、将来のセッションで有害になる可能性はないか？

## auto memory の収集

`auto_memory: true`（デフォルト）の場合、以下の手順で Claude Code 標準の auto memory を収集する:

1. プロジェクトルートの `MEMORY.md` が存在すれば Read で読み込む
2. `~/.claude/projects/<project-hash>/memory/` ディレクトリ内のファイルを Glob で列挙して Read する
3. 収集した memory ファイルの内容を他のデータソースと同様に分析対象に含める

auto memory にはユーザーの判断パターン、フィードバック、プロジェクト知識が蓄積されており、結晶化の最も基本的な入力となる。

## セッションログの収集

`session_logs: true`（デフォルト）の場合、以下の手順で Claude Code のセッション会話ログを収集する:

1. `~/.claude/projects/<project-hash>/` 配下の `*.jsonl` ファイルを Glob で列挙する
2. `crystallization-log.md` の処理位置を参照し、未処理のJSONLファイルのみを対象とする（ファイル名で判定）
3. 各 JSONL ファイルは1行1JSONオブジェクトの形式で、ユーザーの発話、Claudeの応答、ツール呼び出しの全記録を含む
4. 各ファイルは末尾2000行を上限として Read する（offset/limit パラメータを使用）
5. 全ファイルの合計がコンテキストを圧迫する場合は、直近のファイルを優先する

セッションログはNFDの6カテゴリすべてに対応する最もリッチなデータソース:
- `[operational]`: ツール呼び出しと実行結果
- `[reasoning]`: Claudeの判断プロセス
- `[error]`: エラー発生時の対処パターン
- `[context]`: ユーザーの指示やフィードバック
- `[pattern]`: 繰り返し現れるワークフロー
- `[insight]`: 会話中に生まれた気づき

## claude-mem MCP の利用（オプション）

`session_data: true` が設定されており、claude-mem MCP が利用可能な場合:

1. `smart_search` ツールで、プロジェクトに関連する最近のセッションデータを検索する
2. `timeline` ツールで、特定期間のアクティビティを確認する
3. 検索結果を他のデータソースと同様に分析対象に含める

利用できない場合は、警告を出さずに静かにスキップする。

## 注意事項

- 結晶化の8ステップは体系的なフレームワークだが、各ステップでの判断（何を抽出するか、何を残すか）では量より質を重視し、本当に再利用価値のあるパターンのみを選別すること
- 既存のルール/スキルとの整合性を常に確認する
- Gitコマンドは必ず `git -C "<nfd-dir>"` の形式で実行する
